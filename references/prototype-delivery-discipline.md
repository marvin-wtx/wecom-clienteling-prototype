# Prototype Delivery Discipline

A delivery is ready only when scope, business, pages, mobile function, brand design, console, and browser acceptance pass in that order.

- Use common retail structures and plain mock values to create completeness.
- Keep brand-specific claims tied to user sources.
- Implement connected state before decoration.
- Confirm one design direction separately from representative-screen acceptance.
- Build representative screens from the unchanged executable UI kit and a declared page-composition recipe.
- Pass the 390 × 844 Chrome layout probe before asking the user for visual acceptance.
- Use Awesome DESIGN.md only as an optional visual-language source.
- Do not change accepted business structure during visual styling.
- Do not expose untested roles, Journeys, or modules in the console.
- Record one concise Chrome acceptance bound to the final prototype hash.

Tokens and documents support the build; they never prove the experience.

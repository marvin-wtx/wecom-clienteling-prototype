# Operating Model Selection

Choose an operating model from recurring work, not from a brand metaphor.

## Default: core operating workbench

Use this when the brief is sparse or public-only. It should expose the smallest set of connected daily jobs: identify a customer or work item, understand context, take action, and record the next state. It does not prescribe a five-tab IA, a priority queue, or a home-page module order.

## When a specialisation is allowed

A specialised model (appointment preparation, service recovery, event attendance, product interest, manager exception review, or similar) needs at least two independent user-provided operating facts. Valid facts describe real roles, triggers, objects, constraints, outcomes, or existing workflow language. A brand campaign, public store experience, or design reference is not operating evidence.

If the threshold is not met, retain the special idea as a supporting job or omit it. Do not label the product with a poetic operating metaphor.

## Token contract

For every daily job, record:

| Field | Meaning |
| --- | --- |
| `trigger` | What makes this work relevant now? |
| `object` | Which customer, task, appointment, or asset is acted on? |
| `nextAction` | What can the role actually do in the prototype? |
| `outcome` | What route, state, handoff, or write-back proves completion? |
| `evidenceRef` | A claim ID or `generic-baseline` |

Specialisation must record the two qualifying evidence references. If these cannot be named, use `core-operating`.

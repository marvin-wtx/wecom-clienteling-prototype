# Operating Language

Write UI as a frontline advisor speaks while handling work, never as an implementation team describes a system.

For every visible label, ask: **what is happening now, what can the advisor decide or do, and would they naturally say this?** Keep the label only when all three answers are clear.

| Avoid | Use when true |
| --- | --- |
| 最近回写结果 | 今日跟进记录 / 刚刚记录 |
| 今日尚未回写 | 暂时没有新的跟进记录, or omit the section before an action |
| open事项 | 待处理事项 |
| 系统 1v1 | 客户跟进 |
| 最高优先·今日open | 优先处理·今天截止 |

Do not show `open`, `QA`, `token`, `review`, `shell`, `implementation`, evidence status, or system-integration jargon in the mobile product. Use `同步` or `记录` only when the confirmed blueprint actually includes that behavior.

Common structures and mock values may use customer names, membership tiers, transactions, interactions, tasks, appointments, and performance figures when their pages were selected. Mark the product once as demo data. Keep the labels plain and do not describe mock values as the brand's real membership, customer, CRM, store, campaign, or operating rules.

The first viewport is a workbench, not an explanation. Follow the confirmed home selections, show current work and one dominant next action, and avoid a paragraph that narrates a fictional operating scenario.

In `visual-token.json`, record at least three `operatingLanguage.selfCheck` examples with the exact UI text, its work moment, and its decision/action. Record at least two rejected phrases. This is a brief semantic calibration, not another process document.

# Coverage Checklist

## Capabilities

- [ ] Add/binding covered or explicitly out of scope.
- [ ] Client/C360 covered or explicitly out of scope.
- [ ] Task covered or explicitly out of scope.
- [ ] Appointment/invitation covered or explicitly out of scope.
- [ ] Content covered or explicitly out of scope.
- [ ] Dashboard/tracking covered or explicitly out of scope.
- [ ] Transfer/ownership covered or explicitly out of scope.
- [ ] Project-specific integrations identified separately.
- [ ] Opportunity follow-up included only if priority-customer/opportunity lifecycle scope is confirmed.
- [ ] If source material was weak, a baseline framework and assumptions are documented.
- [ ] WeCom mini-program/channel premise is documented.

## Flow Completeness

- [ ] Primary flows have entry, action, result, and exception state.
- [ ] Roles and permissions are visible.
- [ ] FA/SA/BA/advisor terminology matches the industry and client language.
- [ ] Customer/member field vocabulary is confirmed or neutral generic labels are used.
- [ ] Project-specific field names are not used as generic defaults.
- [ ] Customer/contact states are visible.
- [ ] External integration signals and write-backs are clear.
- [ ] WeCom contact state and mini-program card/share preconditions are clear.
- [ ] Search, filter, sort, tabs, page states, and disabled-action reasons are standardized where they appear.
- [ ] Search fields and local/global lookup assumptions are documented.
- [ ] Filter groups, selected states, and no-result states are covered for A-level list pages.
- [ ] Task source, type, target grain, execution channel, completion feedback, and metrics are defined.
- [ ] 1v1, 1vN, Moments, native broadcast, appointment, content, or offline task patterns are separated when in scope.
- [ ] Native WeCom page replicas are included or explicitly deferred.
- [ ] 新建群发 replica includes recipient count, content payload, asset selection entry, send action, frequency note, and return state when in scope.
- [ ] Visual direction, fidelity, and reference sources are documented.
- [ ] Desktop review stage and mobile full-screen mode are both specified when required.
- [ ] Stage controls are hidden in mobile mode.
- [ ] Role switching, free browse, and preset journey behavior are covered when required.
- [ ] Open questions are documented.
